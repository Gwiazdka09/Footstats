## 2. Backfill Neon → Supabase (TYLKO po 1.08 — quota Neon wraca)
> **Merge-safe, NIE nadpisuj:** Supabase ma już świeże wiersze (predykcje od 07-20). Dumpuj data-only i wstaw bez klobrowania.
> Priorytet odzysku: **settled predictions** (licznik → 88 do kalibracji/M1) + konta userów + kupony.
```bash
# a) dump historyczny z Neona (data-only, idempotentnie)
pg_dump "$NEON_URL" --data-only --inserts --on-conflict-do-nothing \
  -t predictions -t coupons -t bankroll_state -t bankroll_history \
  -t users -t ai_feedback \
  > neon_backfill.sql

# b) DRY-RUN: policz ile wierszy w dumpie vs ile już w Supabase (nie wstawiaj na ślepo)
grep -c "INSERT INTO" neon_backfill.sql

# c) restore do Supabase
psql "$SUPABASE_URL" -f neon_backfill.sql
```
> ⚠️ `--on-conflict-do-nothing` działa tylko gdy tabele mają PK/unique na kolizji. Jeśli `predictions` nie ma unique na (team_home,team_away,match_date,ai_tip) → możliwe duplikaty. Wtedy: restore do temp-schematu + `INSERT ... SELECT ... WHERE NOT EXISTS`. **Zawsze porównaj COUNT(*) przed/po.**
> ⚠️ `users` z hasłami (bcrypt hash) — przenoszą się, ale sprawdź czy loginy nie kolidują ze świeżymi kontami Supabase.
